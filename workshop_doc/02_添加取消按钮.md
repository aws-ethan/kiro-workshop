## 问题背景

在开始之前，先了解一下我们要修复什么：

1. 打开浏览器，点击 New Post 菜单进入新建帖子页面
2. 观察当前的 UI，你会看到：
   - 一个帖子内容输入框
   - 一个 "Create Post" 按钮
   - 没有任何取消或返回的方式

> ℹ️ 记住当前 UI 的样子，实现功能后可以对比变化。

问题在于：用户如果改变主意或误点了 "New Post"，没有办法返回 Feed 页面。这是糟糕的用户体验——每个表单都应该有取消选项。

## 你的任务

在 New Post 页面添加一个取消按钮，要求：
- 点击后返回 Feed 页面
- 如果用户已经开始输入内容，弹出确认对话框提醒
- 与现有按钮风格保持一致

## 步骤 1：开始 Vibe 会话

1. 打开 Kiro
2. 打开聊天面板（通常在右侧）
3. 点击顶部的 "+" 图标创建新会话
4. 选择 Vibe 模式

> ℹ️ 为什么用 Vibe 模式？这是一个单组件的简单修改，非常适合 Vibe 模式的对话式开发。

## 步骤 2：编写提示词

以下是一个效果不错的提示词：

> 💬 提示词示例
> ```
> 我想在 New Post 页面添加一个"取消"按钮。点击后应该返回 Feed 页面。如果用户已经在输入框中输入了内容，需要弹出确认对话框提醒草稿将会丢失。
> ```

### 为什么这个提示词有效

| 要素 | 作用 | 重要性 |
|------|------|--------|
| "取消"按钮 | 明确目标 | Kiro 知道要创建什么 |
| New Post 页面 | 指定位置 | Kiro 知道修改哪个组件 |
| 返回 Feed 页面 | 定义行为 | Kiro 知道要实现什么动作 |
| 如果用户已经开始输入 | 边界情况处理 | 防止数据丢失 |
| 弹出确认对话框 | UX 细节 | 指定如何处理边界情况 |

你不需要指定的内容：
- ❌ 文件路径（`frontend/src/pages/CreatePost.tsx`）
- ❌ 实现细节（React Router 的 `useNavigate` hook）
- ❌ 样式细节（按钮 class、颜色）
- ❌ 状态管理（如何追踪用户是否输入）

Kiro 会通过 Steering 文档自动搞定这些。

### 反面示例

太模糊的提示词：
```
加个取消按钮
```
问题：没有指定位置和行为，Kiro 只能猜测。

太详细的提示词：
```
在 CreatePost.tsx 文件中，从 react-router-dom 导入 useNavigate，
添加一个 className 为 "cancel-btn" 的 button 元素，实现一个 onClick
处理函数来检查 content state 是否为空...
```
问题：你在替 Kiro 做事，这违背了 AI 辅助的初衷，还可能与现有模式冲突。

## 步骤 3：审查变更

Kiro 会分析代码库并进行修改。完成后：

1. 在聊天中找到 "View diff" 或 "Show changes" 按钮
2. 点击查看变更内容
3. 仔细检查修改：

审查时需要关注的问题：
- Kiro 是否修改了正确的组件？（应该是 `CreatePost.tsx`）
- 按钮放置的位置是否合理？（应该在 Create Post 按钮附近）
- 草稿检测的实现逻辑是否正确？
- 样式是否与现有按钮一致？
- 是否有意料之外的修改？

## 步骤 4：测试功能

1. 回到浏览器（应该会自动刷新）
2. 进入 New Post 页面

### 测试清单

逐一测试以下场景：

- 空表单测试：在输入框为空时点击 Cancel → 应直接跳转到 Feed（无警告）
- 草稿测试：输入一些文字后点击 Cancel → 应弹出确认对话框
- 确认丢弃：输入文字 → 点击 Cancel → 点击 "OK" → 应跳转到 Feed，文字丢失
- 取消丢弃：输入文字 → 点击 Cancel → 点击 "取消" → 应留在当前页面，文字保留
- 视觉测试：Cancel 按钮是否看起来协调？位置是否合理？样式是否一致？

> ⚠️ 如果遇到问题：
> - 按钮没出现？检查浏览器控制台（F12 → Console），确认开发服务器还在运行，尝试强制刷新（Cmd+Shift+R）
> - 导航不工作？检查控制台的路由错误
> - 样式不对？用后续提示词修正："让取消按钮的样式和 Create Post 按钮保持一致"

## 步骤 5：迭代优化（可选）

对结果不满意？可以用后续提示词调整：

```
把取消按钮放到 Create Post 按钮的左边
```

```
把确认对话框的文案改成"确定要丢弃草稿吗？"
```

```
添加键盘快捷键（Escape 键）来触发取消按钮
```

## 学到了什么

- Vibe 模式很快——从提示词到可用功能只需几秒
- Steering 提供上下文——你不需要解释使用什么框架、组件在哪里、如何设置样式、如何导航
- 迭代很自然——用对话式的后续提示词来优化
- 你始终掌控——在接受变更前审查 diff

## 反思

思考一下刚才发生了什么：
- Kiro 是怎么知道把按钮放在哪里的？它通过 `structure.md` 理解了代码库结构，自动找到了 CreatePost 组件。
- 如果没有 Steering 文档会怎样？Kiro 需要分析多个文件来理解你的框架、路由库、样式方案和文件结构——在开始任务之前就消耗了大量上下文窗口。Steering 消除了这些开销。

## 扩展练习：改善注册体验

### 选项 A：密码验证

注册表单没有告诉用户密码要求，导致注册失败体验很差。

要求：
- 在密码输入框下方显示密码要求
- 用户输入时实时显示验证状态（满足的要求打勾）
- 在所有要求满足前禁用注册按钮

> 💬 提示词示例
> ```
> 在注册表单中添加密码验证。在密码输入框下方显示密码要求：至少 8 个字符、一个大写字母、一个小写字母、一个数字和一个特殊字符。用户输入时实时显示反馈，满足的要求打勾。在所有要求满足之前禁用注册按钮。
> ```

### 选项 B：更好的错误展示

注册失败时，应用显示的是通用的 "An error occurred" 而不是实际的错误信息。

要求：
- 显示实际的错误信息而非通用消息
- 用红色文字清晰展示错误
- 用户开始输入时清除错误信息

> 💬 提示词示例
> ```
> 改善注册表单的错误处理。注册失败时，显示后端 API 返回的实际错误信息（如"Username already exists"），而不是通用的"An error occurred"消息。用红色文字将错误信息醒目地展示在表单上方。用户修改任何字段时自动清除错误信息。
> ```

> ⚠️ Vibe Coding 的局限性：随着功能复杂度增加——需要跨多个元素协调、状态管理和关联逻辑——单个对话式提示词的可靠性会下降，你可能需要多次迭代。这正是 Spec 驱动开发存在的意义。在下一个模块中，你将学习如何通过结构化方式（需求 → 设计 → 实现）来处理复杂的多需求功能。
