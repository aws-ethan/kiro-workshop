## 问题

实现评论功能后，你可能会发现一个问题：新的评论 API 端点返回 404。

这是一个常见的真实场景——AI 生成了新的路由文件和处理逻辑，但没有在主服务器入口文件中注册新路由。部署脚本、服务器启动文件这类"胶水代码"往往不在 AI 的关注范围内。

> ℹ️ 这个练习展示了一个重要原则：AI 生成的代码必须与现有的应用架构集成。理解这些集成点对于成功的 AI 辅助开发至关重要。

## 理解问题

### 步骤 1：查看服务器入口文件

打开 `backend/src/server.js`，查看路由挂载部分：

```javascript
const authRoutes = require('./routes/auth');
const postRoutes = require('./routes/posts');
const userRoutes = require('./routes/users');

// ...

app.use('/auth', authRoutes);
app.use('/users', userRoutes);
app.use('/posts', postRoutes);
```

问题在于：服务器只挂载了三个硬编码的路由模块：`auth`、`posts` 和 `users`。如果 Kiro 在 `routes/` 目录下创建了新的 `comments.js` 路由文件，它不会被自动加载。

### 步骤 2：验证问题

1. 检查 Kiro 是否创建了评论相关的路由文件：

```bash
ls backend/src/routes/
```

如果看到了 `comments.js`（或类似文件），就确认了问题——服务器不知道它的存在。

2. 尝试调用评论 API 验证 404：

```bash
# 先获取一个帖子 ID，然后尝试获取评论
curl -H "Authorization: Bearer <your-token>" http://localhost:3001/comments
```

你应该会收到 404 或 `Cannot GET /comments` 的响应。

## 修复问题

### 方案 A：让 Kiro 修复

使用 Vibe 模式，输入以下提示词：

> 💬 提示词
> ```
> backend/src/server.js 中只挂载了 auth、posts 和 users 三个路由。评论功能的路由文件已经创建但没有被注册。请在 server.js 中添加评论路由的挂载，并确保与现有路由风格一致。
> ```

### 方案 B：手动修复

在 `backend/src/server.js` 中添加评论路由：

```javascript
// 在现有的 require 语句后添加
const commentRoutes = require('./routes/comments');

// 在现有的 app.use 语句后添加
app.use('/comments', commentRoutes);
```

> ℹ️ 具体的路由路径和文件名取决于 Kiro 在 Spec 任务中生成的代码。你可能需要查看实际生成的文件来确认正确的路径。

## 验证修复

1. 保存文件后，nodemon 会自动重启服务器
2. 观察终端输出，确认没有报错
3. 重新测试评论 API：

```bash
# 刷新浏览器，尝试在帖子下添加评论
# 评论应该能正常提交和显示
```

4. 回到浏览器，刷新页面，测试评论功能是否正常工作

## 反思

- Kiro 为什么没有更新 server.js？Kiro 专注于实现功能本身（数据库表、路由处理逻辑、前端组件），但可能遗漏了在入口文件中注册新路由这一步。这是 AI 辅助开发中常见的"最后一公里"问题。
- 如何预防这种情况？考虑在 Steering 文档中添加架构集成信息。一个描述"添加新功能时需要在 server.js 中注册路由"的 Steering 规则可以帮助 Kiro 理解全貌。
- AI 可能遗漏哪些其他集成点？路由注册、中间件配置、数据库迁移、环境变量、前端路由配置等都是 AI 生成的代码可能需要手动集成的领域。

> 💡 进阶思考：你可以创建一个 Agent Hook，在 Kiro 完成任务后自动检查 server.js 是否包含了所有路由文件的引用。这正是下一节 Agent Hooks 要学习的内容。
