## 场景

用户希望通过添加评论来更深入地参与帖子互动。这个功能需要跨整个技术栈进行协调修改——数据库 Schema、后端 API 和前端组件。

## 前置条件

开始之前，确认以下内容：

- [ ] 应用正在运行——前端在 `http://localhost:5174`，后端已启动
  - 测试：能否在 Feed 中看到帖子？
- [ ] 已创建测试帖子——Feed 中至少有 2-3 条帖子
- [ ] Steering 文档已存在——检查 `.kiro/steering/` 下是否有 product.md、tech.md、structure.md

## 步骤 1：开始 Spec 会话

1. 在 Kiro 中，点击聊天面板顶部的 "+" 按钮创建新会话
2. 选择 Spec 模式
3. 输入以下提示词：

> 💬 提示词
> ```
> 我想实现一个新功能：用户可以在 Feed 中对帖子添加评论。
> ```

4. 按回车，等待 Kiro 开始生成规格说明

## 步骤 2：审查需求

Kiro 会在 `.kiro/specs/commenting-feature/`（或类似路径）下生成 `requirements.md` 文件。

### 审查清单

用户故事：
- [ ] 清晰的 "作为[用户]，我想要[目标]，以便[收益]" 格式
- [ ] 覆盖主要用例（添加评论）
- [ ] 覆盖查看评论
- [ ] 处理边界情况（空评论、已删除的帖子等）

验收标准：
- [ ] 具体且可测试（如"用户可以查看帖子的所有评论"）
- [ ] 覆盖正常路径和错误情况
- [ ] 包含 UI/UX 要求
- [ ] 提及认证要求

业务规则：
- [ ] 谁可以评论（仅认证用户？）
- [ ] 评论长度限制
- [ ] 用户能否编辑/删除自己的评论？
- [ ] 审核或过滤规则
好的需求示例：

```markdown
## 用户故事 1：添加评论

作为已登录用户
我想要在 Feed 中的帖子上添加评论
以便我可以参与对话并提供反馈

### 验收标准
- 每条帖子下方显示评论输入框
- 用户必须登录才能评论
- 评论限制在 500 个字符以内
- 不能提交空评论
- 新评论提交后立即显示
- 评论显示作者名称和时间戳
```

> ℹ️ 在实际项目中，你需要仔细审查并完善这些需求。为了节省 Workshop 时间，我们直接使用 Kiro 生成的需求继续。

准备好后，点击 "Move to Design" 进入下一阶段。

## 步骤 3：审查设计

Kiro 会生成 `design.md` 文件，包含技术实现细节。

### 设计审查清单

数据库 Schema：
- [ ] 表名和用途清晰
- [ ] 主键和排序键已定义
- [ ] 属性与需求匹配
- [ ] 常用查询的索引
- [ ] 遵循现有模式

API 端点：
- [ ] RESTful 命名规范
- [ ] HTTP 方法合适（GET、POST、DELETE）
- [ ] 请求/响应格式已定义
- [ ] 错误响应已指定
- [ ] 认证要求已标注

前端组件：
- [ ] 组件层级合理
- [ ] Props 和状态管理已定义
- [ ] 遵循现有 React 模式

需要警惕的问题：
- ❌ 与现有架构不一致
- ❌ 缺少错误处理
- ❌ 没有验证逻辑
- ❌ 组件职责不清晰

> ℹ️ 注意 Kiro 如何自动引用你的技术栈（来自 Steering 文档）。你不需要在提示词中指定这些。

### 基于属性的测试（Property-Based Testing）

在设计阶段，你可能会注意到 Kiro 从需求中提取了"属性"。这是 Kiro 的基于属性测试能力——一种验证实现是否真正符合规格的强大方法。

什么是属性？属性是关于系统行为的通用声明。不是测试具体例子（"用户 A 添加评论 X"），而是表达应该始终成立的规则：

> "对于任何已认证用户和任何帖子，当用户添加评论时，该评论应出现在该帖子上。"

| 传统测试 | 基于属性的测试 |
|---------|--------------|
| 用户添加 "好帖子！" 到帖子 #5 | 对于任意用户和任意帖子，添加任意有效评论都应成功 |
| 测试一个具体案例 | 自动生成数百个测试用例 |
| 受限于作者的想象力 | 发现你想不到的边界情况 |

当 AI 生成实现代码时，基于属性的测试提供了代码确实按规格工作的证据。

准备好后，点击 "Move to implementation plan" 进入任务分解。

## 步骤 4：审查任务

Kiro 会生成 `tasks.md` 文件，将功能拆分为可实现的任务块。

注意，和非 AI 辅助的项目一样，Kiro 会建议将某些任务标记为可选，取决于你是否想实现快速 MVP。

每个任务都会有一个 "Start task" 链接。

## 步骤 5：实现所有任务

现在通过逐个完成任务来构建完整功能：

1. 在 `tasks.md` 文件中，找到第一个任务
2. 点击旁边的 "Start task"
3. 等待 Kiro 完成任务（通常每个任务 30-60 秒）
4. 审查 Kiro 所做的修改
5. 继续下一个任务，重复以上步骤

> ℹ️ 有些任务可能包含子任务。启动父任务时，Kiro 会自动排队所有子任务。你也可以单独启动每个子任务以获得更精细的控制。

> ⚠️ 实现所有任务通常需要 5-10 分钟。按顺序完成每个任务，Kiro 会显示进度。

## 步骤 6：测试功能

### 初始设置

1. 确认前端正在运行
2. 刷新浏览器以加载新代码
3. 进入 Feed 页面——你应该能看到已有的帖子

### 测试脚本

逐一验证以下功能：

测试 1：查看评论区
- [ ] 查看 Feed 中的帖子，评论区应出现在帖子下方
- [ ] "添加评论" 输入框可见
- [ ] 初始状态无评论（或显示已有评论）

测试 2：添加第一条评论
- [ ] 在输入框中输入 "这是我的第一条评论！"
- [ ] 点击提交
- [ ] 评论立即出现在帖子下方，显示用户名和时间戳
- [ ] 输入框在提交后清空

测试 3：添加多条评论
- [ ] 在同一帖子上再添加 2-3 条评论
- [ ] 所有评论按顺序显示

测试 4：持久化检查
- [ ] 刷新页面（Cmd+R 或 Ctrl+R）
- [ ] 所有评论仍然显示，顺序不变

测试 5：多帖子测试
- [ ] 在不同帖子上添加评论
- [ ] 每条帖子只显示自己的评论，不会混淆

测试 6：空评论验证
- [ ] 尝试提交空评论
- [ ] 应显示错误信息或禁用提交按钮

测试 7：长评论
- [ ] 输入超过 500 个字符的评论
- [ ] 应显示字符计数器（如已实现）
- [ ] 超出限制时禁用提交按钮（如已实现）

> ✅ 你已经成功使用 Spec 驱动开发实现了一个完整功能：后端新增了数据库表和 API 端点，前端新增了评论组件，包含错误处理和验证逻辑。

## 排错

- 评论添加后不显示？检查浏览器控制台的 API 错误（F12 → Console），确认后端正在运行
- 前端报错？清除浏览器缓存并强制刷新（Cmd+Shift+R），检查 `npm run dev` 是否正常运行

## 反思

思考一下你刚才完成的工作：

- 与取消按钮练习对比：有什么不同？取消按钮是单组件修改，评论功能需要协调数据库、API 和 UI 的跨文件修改。Spec 模式为这种复杂性提供了结构。
- 结构化方法如何帮助处理复杂性？需求给了你明确的目标，设计让你在编码前审查架构，任务将工作拆分为可管理的小块。
- 如果用 Vibe 模式能做到吗？技术上可以，但你需要很多提示词和仔细的协调，可能会遗漏边界情况或在各层之间产生不一致。Spec 模式确保不会遗漏。
