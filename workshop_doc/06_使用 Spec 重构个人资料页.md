## 场景

当前的个人资料页（Profile）功能比较基础，UI 也比较粗糙。项目中已经有一份详细的设计语言文档（`DESIGN_LANGUAGE.md`），但 Kiro 默认不会参考它。

在这个练习中，你将学习：
1. 创建条件触发的 Steering 文件，让设计文档在相关时自动加载
2. 用 Spec 模式驱动一个有设计规范约束的 UI 重构任务

这是 Steering + Spec 的高级组合用法。

## 前置条件

- [ ] 应用正在运行
- [ ] `DESIGN_LANGUAGE.md` 存在于项目根目录
- [ ] 已完成前面的 Steering 文档生成（`.kiro/steering/` 下有文件）

## 步骤 1：创建条件触发的 Steering 文件

### 为什么需要条件触发

默认的 Steering 文件（`product.md`、`tech.md`、`structure.md`）每次对话都会加载。但设计语言文档只在修改 UI 时才需要——如果每次都加载，会浪费上下文窗口。

Kiro 支持三种 Steering 加载方式：
- `always`（默认）：每次对话都加载
- `fileMatch`：当特定文件被读入上下文时才加载
- `manual`：用户通过 `#` 手动引用时才加载

我们要创建一个 `fileMatch` 类型的 Steering，当 Kiro 读取 `.tsx` 或 `.css` 文件时自动加载设计规范。

### 创建 Steering 文件

1. 在 Kiro 中，使用 Vibe 模式输入：

> 💬 提示词
> ```
> 帮我在 .kiro/steering/ 下创建一个名为 design.md 的 Steering 文件。要求：
> 1. 使用 fileMatch 条件触发，当读取 .tsx 或 .css 文件时自动加载
> 2. 内容引用项目根目录的 DESIGN_LANGUAGE.md
> 3. 告诉 Kiro 在修改 UI 组件时必须遵循设计语言规范
> ```

2. 或者手动创建 `.kiro/steering/design.md`，内容如下：

```markdown
---
inclusion: fileMatch
fileMatchPattern: "**/*.tsx,**/*.css"
---

# 设计语言规范

在修改任何 UI 组件或样式时，必须遵循项目的设计语言规范。

设计语言文档位于：#[[file:DESIGN_LANGUAGE.md]]

关键要求：
- 使用规定的色彩系统（主色调 #8b5cf6，背景色 #f5f8fa 等）
- 按钮使用全圆角（border-radius: 9999px）
- 遵循间距系统（xs: 4px, sm: 8px, md: 16px, lg: 24px）
- 使用系统字体栈
- 卡片使用 8px 圆角和 1px 边框
- 确保响应式设计（移动端 ≤768px，桌面端 ≥769px）
```

### 验证 Steering 加载

1. 打开一个新的 Vibe 聊天会话
2. 输入：

```
看一下 Profile.tsx 的代码，告诉我它的 UI 结构
```

3. 观察聊天顶部——当 Kiro 读取 `.tsx` 文件时，应该会显示加载了 `design.md` Steering 文件
4. Kiro 的回复应该会参考设计语言规范来评价当前的 UI

## 步骤 2：用 Spec 模式重构个人资料页

### 开始 Spec 会话

1. 点击 "+" 创建新会话
2. 选择 Spec 模式
3. 输入以下提示词：

> 💬 提示词
> ```
> 重构个人资料页（Profile），使其符合项目的设计语言规范，并提升用户体验。具体要求：
> 
> 1. 个人资料头部：使用卡片样式展示用户信息，包含头像占位符、显示名称、用户名、简介、关注者/关注中统计
> 2. 帖子列表：使用与 Feed 页面一致的帖子卡片样式
> 3. 编辑资料：改善编辑表单的布局和交互，添加取消确认
> 4. 所有样式必须遵循 DESIGN_LANGUAGE.md 中的设计规范
> ```

### 审查需求

Kiro 生成 `requirements.md` 后，重点检查：

- [ ] 是否引用了设计语言规范中的具体样式要求
- [ ] 用户故事是否覆盖了查看资料、编辑资料、查看帖子等场景
- [ ] 验收标准是否包含具体的视觉要求（颜色、间距、圆角等）
- [ ] 是否考虑了响应式设计

> 💡 注意 Kiro 如何利用条件触发的 Steering 文件——它应该自动引用 `DESIGN_LANGUAGE.md` 中的规范，而不需要你在提示词中重复这些细节。

准备好后，点击 "Move to Design"。

### 审查设计

在 `design.md` 中，重点关注：

- [ ] 组件结构是否合理（ProfileHeader、ProfilePosts、EditProfileForm 等）
- [ ] 样式方案是否使用了设计语言中定义的 CSS 变量
- [ ] 是否复用了现有的组件模式（如帖子卡片）
- [ ] 响应式断点是否与设计语言一致（768px、1200px）

准备好后，点击 "Move to implementation plan"。

### 实现任务

1. 逐个完成 `tasks.md` 中的任务
2. 每个任务完成后，在浏览器中检查视觉效果
3. 对比 `DESIGN_LANGUAGE.md` 中的规范，确认样式一致性

## 步骤 3：视觉验证

### 检查清单

完成所有任务后，逐项验证：

色彩系统：
- [ ] 卡片背景为白色（#ffffff）
- [ ] 页面背景为浅灰色（#f5f8fa）
- [ ] 按钮使用紫色主色调（#8b5cf6）
- [ ] 文字层次清晰（主要 #14171a，次要 #657786）

组件样式：
- [ ] 个人资料卡片有 8px 圆角和 1px 边框
- [ ] 按钮为全圆角（pill 形状）
- [ ] 编辑/关注按钮样式与设计语言一致
- [ ] 表单输入框有 4px 圆角

布局：
- [ ] 桌面端布局合理，内容不会过宽
- [ ] 缩小浏览器窗口到手机宽度，布局自适应
- [ ] 间距符合设计语言的间距系统

交互：
- [ ] 编辑资料表单可以正常保存和取消
- [ ] 关注/取消关注按钮状态切换正确
- [ ] 帖子列表正常显示

## 步骤 4：迭代优化（可选）

如果某些样式不符合设计规范，用 Vibe 模式快速修正：

```
Profile 页面的按钮圆角应该是 9999px（全圆角），不是 8px，请按照 DESIGN_LANGUAGE.md 修正
```

```
个人资料卡片的间距不对，应该使用 1.5rem 的内边距，参考设计语言中的卡片规格
```

> 💡 因为 `design.md` Steering 是条件触发的，当 Kiro 读取 `.tsx` 文件时会自动加载设计规范，所以你的修正提示词不需要重复解释设计要求。

## 反思

### Steering 的分层策略

通过这个练习，你体验了 Steering 的分层加载：

- `product.md`、`tech.md`、`structure.md`：始终加载，提供基础上下文
- `design.md`：条件触发，只在修改 UI 时加载，避免浪费上下文

这种策略在大型项目中尤其重要——你可以为不同领域创建专门的 Steering 文件：
- `api-standards.md`：修改 API 路由时加载
- `security.md`：修改认证相关代码时加载
- `testing.md`：编写测试时加载

### Spec + Steering 的协同效应

- Steering 提供"怎么做"的规范（设计语言、编码标准）
- Spec 提供"做什么"的结构（需求 → 设计 → 任务）
- 两者结合，AI 既知道目标，又知道约束，生成的代码质量更高

### 与纯 Vibe 模式的对比

如果用 Vibe 模式做这个重构：
- 你需要在提示词中详细描述每个样式要求
- 多组件修改容易遗漏或不一致
- 没有结构化的审查环节，容易偏离设计规范

Spec 模式让你在编码前就审查了设计方案，确保实现与规范一致。

## 总结

恭喜你完成了整个 Workshop。回顾一下你学到的 Kiro 核心能力：

| 模块 | 学到的能力 | 适用场景 |
|------|-----------|---------|
| Steering 文档 | 持久化项目上下文 | 所有场景——消除重复解释 |
| Vibe 模式 | 对话式快速开发 | 单组件修改、快速修复 |
| Spec 模式 | 结构化功能开发 | 跨文件复杂功能 |
| 问题排查 | AI 集成点意识 | 路由注册、配置遗漏 |
| Agent Hooks | 自动化工作流 | 代码检查、测试、审查 |
| 条件 Steering | 按需加载上下文 | 设计规范、API 标准 |

> 💡 带走的关键认知：Kiro 不是替代开发者，而是增强开发者。你始终掌控方向（需求审查、设计审查），Kiro 加速执行。理解 AI 的能力边界（如集成点遗漏）和如何通过 Steering、Hooks 来弥补，是高效使用 AI 辅助开发的关键。
